# RAG 代码扩展需求分析

## 核心目标

当用户通过 @ 指定一个或多个 Python 文件时，本技能会：
1. 理解代码在 RAG 流程中的定位
2. 提出**动手练习需求**，帮助用户通过改代码来理解 RAG 概念
3. 生成需求文档到 `req` 目录

## 练习需求设计原则

| 原则 | 说明 |
|------|------|
| **小步快跑** | 每个需求工作量 15-30 分钟内可完成 |
| **概念驱动** | 每个练习背后对应一个 RAG 核心概念 |
| **可验证** | 有明确的运行结果来验证是否成功 |
| **层层递进** | 需求之间有难度梯度 |

## 工作流程

### 第一步：读取并分析代码

### 第二步：生成需求文档

保存位置：代码同级 `req/` 目录，命名 `<被分析文件名>_需求.md`

### 第三步：需求文档结构

```markdown
# [文件名] 动手练习清单

## RAG 流程定位
当前代码属于 RAG 的哪个环节？
- 输入：
- 处理：
- 输出：

## 练习任务

### 练习 1：[任务名称]
**学习目标**：理解 [RAG 概念]

**任务描述**：
大白话描述要做什么

**实现思路**：
```python
# 关键代码提示
```

**验证方法**：
运行代码后看到什么结果说明成功了

**扩展思考**（可选）：
- 如果改成 xxx 参数会怎样？
- 这个方案有什么局限性？

### 练习 2：...

## 相关知识点
| 概念 | 解释 |
|------|------|
| XXX | YYY |
```

## 需求描述要求

每个练习必须包含：
1. **学习目标**：明确练习对应的 RAG 概念
2. **任务描述**：用大白话说清楚要做什么
3. **关键代码提示**：给出核心代码片段或伪代码
4. **验证方法**：如何验证练习成功
5. **扩展思考**：引发进一步思考的问题

## 注意事项

- 输出语言与用户一致
- req 目录不存在时自动创建
- 练习要简单实用，避免复杂工程
- 重点帮助理解 **RAG 知识**，而非写漂亮代码
